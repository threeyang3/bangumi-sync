# 更新项目

当用户发送"更新项目"时，依次执行以下步骤：

## 版本号规则

- **小版本更新**（默认）：版本号增加 0.1（如 3.2.0 → 3.3.0）
  - 本地源代码可以覆盖同一大版本的前几个小版本
  - 直接更新现有文件

- **大版本更新**（需特别声明"更新大版本"）：大版本号 +1，小版本号归零
  - 新建一套本地文件（如 v4/ 目录）
  - 需要创建新的目录结构和配置文件

## 步骤 1: 更新版本号

1. 读取 `v3/manifest.json` 获取当前版本号
2. 计算新版本号：
   - 小版本更新：`3.2.0` → `3.3.0`
   - 大版本更新：`3.2.0` → `4.0.0`
3. 更新 `v3/manifest.json` 中的版本号

## 步骤 2: 更新 README.md

README.md 结构要求：
1. **主要功能** - 描述所有已实现的功能（作为主体内容）
2. **最新更新** - 只描述最新一次更新的内容，不需要历史版本列表

更新步骤：
1. 读取 `CLAUDE.md` 了解最新功能
2. 将新功能合并到"主要功能"部分
3. 更新"最新更新"部分，只保留最新版本的更新内容

## 步骤 3: 更新发布 Release

1. 执行 `npm run build:v3` 编译 V3 版本
2. 复制 `v3/main.js`、`v3/manifest.json`、`styles.css` 到 `release/` 目录
3. 在 `release/` 目录创建 `bangumi-sync-v{版本号}.zip` 压缩包

## 步骤 4: 提交并推送到 GitHub

1. 检查 git 状态，查看所有修改的文件
2. 添加修改到暂存区（**排除旧版本目录 v1/、v2/**）
3. 创建提交，提交信息格式：`chore: bump version to {版本号}` 或 `feat: {功能描述}`
4. 推送到远程仓库

```bash
git status
git add . ':!v1/' ':!v2/'
git commit -m "chore: bump version to {版本号}"
git push origin main
```

**注意**：V1、V2 等旧版本文件夹仅保留在本地，不推送到 GitHub。确保 `.gitignore` 中包含：
```
v1/
v2/
```

## 步骤 5: 更新 CLAUDE.md

1. 确认版本历史部分已更新
2. 确认功能描述与当前实现一致
3. 确认项目结构文档反映最新文件

## 执行命令

```bash
# 编译
npm run build:v3

# 复制文件到 release
cp v3/main.js release/
cp v3/manifest.json release/
cp styles.css release/

# 创建 zip
cd release && powershell -Command "Compress-Archive -Path main.js,manifest.json,styles.css -DestinationPath bangumi-sync-v{版本号}.zip -Force"
```

## 注意事项

- 确保所有代码修改已完成并编译通过
- 确保版本号已更新
- release 目录中的三个文件（main.js、manifest.json、styles.css）时间戳应为最新
